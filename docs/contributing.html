Contributing

How to contribute to Ledewire — from branch conventions and commit messages to the pull request process and code review expectations.

Workflow Overview

  1. Create a branch off main
  2. Make your changes, with tests
  3. Open a pull request against main
  4. Address review feedback
  5. Merge after approval

Branch Conventions

Use the following prefixes for branch names:

  • feat/short-description — new features
  • fix/short-description — bug fixes
  • chore/short-description — maintenance, tooling, refactors
  • docs/short-description — documentation changes
# Examples
git checkout -b feat/user-authentication
git checkout -b fix/login-redirect-loop
git checkout -b docs/update-architecture-overview

Commit Messages

Follow Conventional Commits: 

type(scope): short description in imperative mood

# Examples
feat(auth): add OAuth2 login flow
fix(api): return 404 for missing resources instead of 500
chore(deps): bump @types/node to 20.x
docs(readme): update local setup instructions

Pull Requests

Before opening a PR, make sure:

  • All tests pass locally — npm test
  • Linting passes — npm run lint
  • Your branch is up to date with main
  • The PR description explains what changed and why

Keep PRs small and focused. Large, sprawling PRs are hard to review and slow to merge. If your change touches many unrelated areas, consider splitting it into sequential PRs.

PR Description Template

## What changed
[Brief summary of what this PR does]

## Why
[Context: issue reference, business reason, or problem solved]

## How to test
[Steps for the reviewer to verify the change works]

## Screenshots (if applicable)

Code Review

Reviews should be constructive and focused on the code. As an author, be open to feedback. As a reviewer:

  • Approve when the change is good enough — don’t block on style nits
  • Use blocking comments only for correctness issues
  • Use suggestions for style and preference feedback
  • Aim to review within one business day
  • Ask questions before assuming intent

Coding Standards

We use ESLint with Prettier for consistent formatting. The config lives in .eslintrc.js and .prettierrc at the repo root. Run npm run lint:fix to auto-fix most issues before committing.

Questions?

If you’re unsure about an approach, ask in the team’s engineering channel before investing time in a direction that might not align. It’s always better to ask early than to rework late.
