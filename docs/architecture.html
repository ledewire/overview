Architecture Overview

A high-level view of how Ledewire is designed — the services, boundaries, and the principles that guide how we build.

Design Principles

These principles guide every significant technical decision in Ledewire:

  • Simplicity over cleverness. We prefer clear, boring code over clever abstractions.
  • Explicit over implicit. Behavior should be visible and easy to trace.
  • Service boundaries matter. Each service owns its data and exposes clean interfaces.
  • Observability first. If we can’t measure it, we can’t improve it.
  • Build for change. Defer decisions that can be deferred; avoid over-engineering early.

Services

Ledewire is composed of several services, each with a focused responsibility. See Service Details for per-service documentation.

Service Responsibility Owner
api-gateway Routing, auth enforcement, rate limiting
core-service Primary business logic
data-service Data persistence & retrieval
worker Async job processing

Data Flow

Requests enter through the API gateway, which handles authentication and routes to the appropriate service. Core business logic lives in the core service, which delegates persistence to the data service. Long-running operations are enqueued and processed asynchronously by the worker. 

Client
  └── API Gateway        (auth, routing)
        ├── Core Service   (business logic)
        │     └── Data Service  (persistence)
        └── Worker         (async jobs)
              └── Data Service  (persistence)

Tech Stack

Key technology decisions are documented in the ADRs. This is a high-level summary:

  • Runtime: Node.js
  • Language: TypeScript
  • Database: PostgreSQL
  • Queue: Redis / BullMQ
  • Infrastructure: Docker, GitHub Actions

For the rationale behind these choices, see the Architecture Decision Records.
